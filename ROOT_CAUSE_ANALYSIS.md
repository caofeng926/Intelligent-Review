# 根因分析与预防指南

> 基于「医保智审规则库」四轮代码审查的深度复盘
>
> 分析日期：2026-07-07 | 对应审查报告：CODE_REVIEW.md / R2 / R3 / R4

## 一、问题不在于代码，在于开发模式

四轮审查发现了 19 项技术债 + 4 项回归 bug。表面看是"代码质量参差不齐"，
但深挖根因后发现：**绝大多数问题源于同一个结构性缺陷——脚本堆砌式开发。**

### 关键证据

| 指标 | 数据 | 含义 |
|------|------|------|
| `_*.py` 一次性脚本 | **102 个**（根目录 88 + webapp/ 14） | 开发者的"影子版本控制" |
| `_parse_v1.py` → `_parse_v8.py` | 8 个版本 | 迭代靠复制文件，不靠 git commit |
| `_check_db.py` / `_check_db2.py` / `_check_db_path.py` | 3 个变体 | 同一功能重复创建 |
| `_patch_xlsx1` / `_v2` / `_main` / `_xlsx2` / `_xlsx3` | 5 个变体 | 补丁脚本无版本管理 |
| 质量工具引入时机 | 第 17 次 commit（共 36 次） | 前 47% 的代码无任何质量门禁 |
| 单元测试引入时机 | 第 34 次 commit | 测试在项目末期才补 |
| commit 消息带 BOM (`﻿`) | 5 条 | Windows 编码问题渗透到 git 历史 |

## 二、三大根因

### 根因 1：脚本堆砌开发模式

**现象**：开发者的真实工作流是"复制文件 → 修改 → 测试 → 丢弃"，而非
"commit → 迭代 → 验证"。102 个 `_` 前缀脚本就是"影子版本控制"的遗迹。

**因果链**：

```
脚本堆砌模式
  ├→ 迭代靠复制文件 (_parse_v1→v8)
  │    └→ 无 git 历史 / 无 diff / 无回退
  │    └→ 旧版本变成死代码，无人清理
  │
  ├→ 脚本无质量要求 (不需要 lint/test)
  │    └→ 硬编码密码成为习惯 (TD-01)
  │    └→ except: pass 成为习惯 (TD-03)
  │    └─→ 无输入验证成为习惯 (TD-07)
  │
  ├→ 脚本晋升为正式代码时坏习惯带入
  │    └→ _wait_upload.py 带密码进 git (TD-01)
  │    └→ clean_drug_detail.py 用 total_changes (TD-02)
  │
  └→ 清理永远不发生
       └→ 102 个脚本堆积，项目根目录混乱
       └→ 新人无法区分"正式代码"和"废弃脚本"
```

**为什么这是根因**：脚本不需要质量门禁（lint、测试、CI），因为它们是
"一次性的"。但开发者在这种模式下养成的习惯——硬编码、裸 except、
不验证输入——会本能地带入正式代码。`except: pass` 不是"忘了加错误
处理"，而是"脚本里从来不需要错误处理"。

### 根因 2：质量门禁缺失

**现象**：pre-commit、flake8、CI、测试——全部在第一轮审查后才补上。
在此之前，所有代码都在"裸奔"状态。

**因果链**：

```
质量门禁缺失
  ├→ 无 lint
  │    └→ except: pass 未被拦截 (TD-03)
  │    └→ import 顺序混乱
  │    └→ 未使用的导入未被检测 (NEW-04 的死导入)
  │
  ├→ 无 CI
  │    └→ 密钥泄露未被自动扫描 (TD-01)
  │    └→ .bak 文件未拦截
  │
  ├→ 无测试
  │    └→ total_changes bug 未被发现 (TD-02)
  │    └→ 重构后回归未被捕获 (NEW-04)
  │    └─→ fts_query 中文处理被绕过未被发现 (NEW-07)
  │
  └→ 无冒烟测试
       └→ admin 搜索 NameError 存活 2 轮 (NEW-04)
       └→ "能 import 就行"成为验收标准
```

**为什么这是根因**：人工审查（即使做了 4 轮）能发现"已经存在的"问题，
但无法替代自动化门禁"阻止问题进入"。如果没有 CI，每次重构都像在
雷区走路——你不知道哪里会炸。

### 根因 3：单人开发无交叉审查

**现象**：一个人写所有代码，没有 PR 流程，没有第二双眼睛。

**因果链**：

```
单人开发无交叉
  ├→ 自我审查有盲区
  │    └→ 导入了 fts_query 但调用 _admin_fts_query (NEW-04)
  │    └→ 建了 query_utils 但 _code_search 重复实现 (NEW-06)
  │    └→ 写了 detect_mode 但 rules.py 又抄一遍 (NEW-05)
  │
  ├→ "以后再修"= 以后不修
  │    └→ TD-04 5 行注释拖 4 轮
  │    └→ NEW-04 2 行改动拖 2 轮
  │    └─→ 没有人追问"这个 blocker 修了没"
  │
  └→ 优先级偏差
       └→ 大重构（1115→372 行）优先于小修补
       └→ "有成就感的事"优先于"该做的事"
```

**为什么这是根因**：人在写代码时的心理状态和审查代码时不同。写代码时
关注"让它跑起来"，审查时关注"它真的对吗"。同一个人隔一段时间再审查
自己的代码，效果远不如两个人交叉审查——因为你的思维模式不会自动切换。

## 三、根因之间的协同效应

三个根因不是独立的，它们互相强化：

```
脚本堆砌模式 ──→ 不需要质量门禁 ──→ 质量门禁缺失
     ↑                                    │
     │                                    │
     │    ┌───────────────────────────────┘
     │    │
     │    ↓
     │  无 CI 拦截 ──→ 问题积累 ──→ 单人难以全面清理
     │                                    │
     │                                    ↓
     └── 习惯带入正式代码 ←── 无审查发现 ←──┘
```

这形成了一个**恶性循环**：脚本模式导致不需要质量工具 → 没有质量工具
导致问题积累 → 问题积累导致单人难以清理 → 继续用脚本模式快速迭代 →
更多脚本 → 更多坏习惯带入正式代码。

## 四、今后项目的预防方案

### 预防 1：消灭"脚本堆砌"模式

| 做法 | 具体操作 |
|------|----------|
| 所有代码走 git | 一次性脚本也 commit，用 `chore/scratch-xxx` 分支 |
| 脚本放 `scripts/` 目录 | 不允许在项目根目录创建 `_*.py` |
| 脚本也要有基本质量 | `argparse` 接参数，不硬编码；`try/except` 不裸 pass |
| 定期清理 | 每月删一次 `scripts/` 下的废弃脚本 |
| 迭代用 git 不用复制 | `_parse_v1.py` → `_parse_v2.py` 改为 `git commit` 迭代 |

**核心原则**：如果一个脚本重要到需要保存，它就重要到需要版本管理。
如果不需要保存，就写完用完即删。

### 预防 2：项目第一天就建质量门禁

```bash
# 项目初始化脚本（Day 1 执行）
pre-commit install                    # 提交前检查
# .flake8 — 语法检查
# .pre-commit-config.yaml — 密钥扫描 + .bak 拦截 + isort
# requirements-dev.txt — 开发依赖
# .github/workflows/ci.yml — PR 时自动 lint + test
# tests/ — 至少一个冒烟测试
```

**关键**：不是"等项目成型再补"，而是"第一行代码就有门禁"。因为
前 47% 的代码无门禁 = 这部分代码的债务需要 4 轮审查才能发现。

### 预防 3：强制交叉审查

| 场景 | 做法 |
|------|------|
| 团队开发 | PR 必须至少 1 人 approve 才能合并 |
| 单人项目 | 改完代码隔 2 小时再自审（切换上下文） |
| 单人项目 | 用 AI 辅助审查：`grep "导入但未使用的函数"` |
| 重构时 | 删/改函数 → 全局 grep 调用点 → 跑冒烟测试（强制清单） |

### 预防 4：冒烟测试作为合并条件

```python
# tests/test_smoke.py — 最小冒烟测试
def test_app_starts(client):
    """应用能启动"""
    assert client.get("/").status_code == 200

def test_search_works(client):
    """搜索不崩"""
    assert client.get("/search?q=test").status_code == 200

def test_admin_search(client):
    """管理后台搜索不崩——专门防 NEW-04 这类回归"""
    assert client.get("/admin/search?q=test").status_code == 200
```

NEW-04 这种 NameError，一个 HTTP 请求就能暴露。如果合并前跑冒烟测试，
它根本不可能进 main 分支。

### 预防 5：技术债看板 + 严重度优先

```
┌─────────────────────────────────────┐
│  技术债看板                          │
├──────────┬──────────┬───────────────┤
│ 🔴 当天  │ 🟡 本周  │ 💭 有空再做   │
│          │          │               │
│ NEW-04   │ TD-07    │ TD-12         │
│ TD-04    │ NEW-05   │               │
│          │ NEW-06   │               │
└──────────┴──────────┴───────────────┘
```

**铁律**：🔴 blocker 当天清零，不管几行。2 行和 200 行一样会炸。

### 预防 6：Windows 编码环境治理

```bash
# .editorconfig — 强制 UTF-8
[*.py]
charset = utf-8
end_of_line = lf

# git config
git config --global core.quotepath false
git config --global i18n.commitencoding utf-8

# Python 启动加 -X utf8
python -X utf8 -m webapp.app
```

TD-04 的 5 处乱码不是"写错了"，而是 Windows GBK 环境与 UTF-8 源文件
冲突的系统性问题。不治理环境，修了这 5 处还会有下一批。

## 五、预防方案检查清单

新项目启动时，逐项打勾：

```
□ git init + .gitignore（排除 .env / __pycache__ / *.db）
□ pre-commit install
□ .flake8 配置
□ .pre-commit-config.yaml（flake8 + isort + detect-secrets）
□ requirements-dev.txt
□ .github/workflows/ci.yml（PR 时 lint + test）
□ tests/test_smoke.py（至少 3 个冒烟测试）
□ scripts/ 目录（所有一次性脚本放这里，不放根目录）
□ .editorconfig（charset = utf-8）
□ CODE_REVIEW.md（审查标准，不用完美，有就行）
```

新功能开发时，逐项打勾：

```
□ 创建分支：feat/fix/refactor-<简述>
□ 写代码前先 grep 现有工具函数（避免重复实现）
□ 删/改函数后 grep 全项目调用点
□ 提交前跑 pre-commit
□ 提交前跑冒烟测试
□ 创建 PR（使用模板）
□ 🔴 blocker 当天修
```

## 六、一句话总结

**这个项目的所有问题，都可以追溯到一句话：用写脚本的方式写产品代码。**

脚本不需要版本管理、不需要质量门禁、不需要测试、不需要审查——
但产品代码需要。当开发者用脚本习惯写产品代码时，所有质量保障机制
都天然缺位，问题就会像滚雪球一样积累。

预防的核心不是"写更好的代码"，而是"换一种工作模式"：
**从 Day 1 开始，把每一行代码都当作产品代码对待。**
